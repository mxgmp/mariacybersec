Detecting Brute Force Attacks with Splunk

Brute force attacks aren’t just random password guesses—they are high-volume authentication failures that leave a clear trail in system logs. In this project, I built a Splunk workflow to identify brute force activity, create automated alerts, and visualize the results in a dashboard.

This was my first hands-on Splunk project, and here’s how I did it step by step:

Step 1: Ingest Authentication Logs

Before detection, Splunk needs the right data. I ingested Linux authentication logs (linux_auth_demo) into a dedicated index called lab.

[Screenshot: Data Input / Files & Directories screen]

Step 2: Search for Failed Logins

The raw logs contain both successful and failed attempts. To isolate brute force behaviour, I searched for events with the keyword "Failed password". 

index=lab sourcetype=linux_auth_demo "Failed password"

[Screenshot: Failed login events in Splunk Search & Reporting]

Step 3: Identify Suspicious Spikes

I grouped failed logins by source IP (src) to detect bursts over time: 

index=lab sourcetype=linux_auth_demo "Failed password"
| timechart span=5m count by src

[Screenshot: timechart showing spikes by IP]

Step 4: Create a Real-Time Alert

Detecting is good, but alerts are what analysts rely on. I converted the search into an alert:

  • Type: Real-time
  • Trigger: If failed logins exceed 5 in 5 minutes
  • Throttle: By source IP (avoid spam)
  • Severity: High

[Screenshot: Alert configuration panel]

Step 5: Build a Dashboard Panel

I visualized the results in a Splunk dashboard with:

  • X-axis: Time
  • Y-axis: Failed login attempts
  • Split by: Source IP

[Screenshot: Splunk dashboard panel]

Step 6: Reduce False Positives

Not every spike is malicious. I filtered out service accounts and test data to keep focus on real brute force attempts.

[Screenshot: SPL query with filters]

Conclusion

This project gave me hands-on experience with Splunk search, alerting, and dashboards. I learned that brute force attacks are noisy, and without the right filtering, alerts can overwhelm analysts.
