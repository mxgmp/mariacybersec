Detecting Brute-force Attacks with Splunk

High-volume authentication failures leave a very loud trail. In this lab I built a clean Splunk workflow to spot brute-force activity, alert on it in (near) real-time, and visualise it in a minimal dashboard.

1 Ingest authentication logs

First, I indexed Linux auth logs (linux_auth_demo) into a dedicated index called lab to keep test data isolated.

Menu path: Settings → Data inputs → Files & Directories

2 Search for failed logins

I started simple: look for “Failed password”.

index=lab sourcetype=linux_auth_demo "Failed password"

3 Identify suspicious spikes

Group by source IP (src) over time to spot bursts:

index=lab sourcetype=linux_auth_demo "Failed password"
| timechart span=5m count by src

4 Create a real-time alert

I converted the search into an alert so noisy spikes get surfaced automatically.

  • Type: Real-time (per-result, throttled)
  • Trigger: IP with multiple failures within 5 minutes
  • Throttle: by src for 5 minutes (prevents spam)
  • Severity: High

5 Build a dashboard panel

Quick panel to show the “noisiest” IPs:

  • X-axis: Time
  • Y-axis: Failed login attempts
  • Split by: src

6 Reduce false positives

Filter out test/service accounts so only real brute-force behaviour remains:

index=lab sourcetype=linux_auth_demo "Failed password"
| search user!=svc_* user!=test*
| timechart span=5m count by src

Conclusion. Clean pipeline: ingest → search → aggregate → alert → dashboard → tune. All queries and config screenshots are on this page. A short video recap lives at @maria.cybersec.
